### Non-conformités détectées

**[NC] 7.5 — Messages de statut restitués par les technologies d'assistance**
- **Élément concerné :** Formulaires de création de contact, d'ajout d'interaction, et bouton "Fait" sur les relances
- **Problème :** Après soumission réussie d'un formulaire (création contact, ajout interaction, marquer relance comme faite), aucun message de statut avec `role="status"` ou `aria-live="polite"` n'est restitué. L'utilisateur de lecteur d'écran n'est pas informé du résultat de son action (la page se recharge silencieusement via revalidatePath ou redirige).
- **Correction :** Ajouter un `<div role="status">` qui annonce le succès après chaque action (ex : "Interlocuteur créé", "Interaction enregistrée", "Relance marquée comme faite").
- **Priorité :** 🟠 Majeur

---

**[NC] 8.6 — Titre de page pertinent**
- **Élément concerné :** Balise `<title>` des pages enfants
- **Problème :** Les titres de pages enfants n'incluent pas le nom du site. Exemples : "Nouvel interlocuteur" au lieu de "Nouvel interlocuteur - Suivi ASN", "Alice Durand" au lieu de "Alice Durand - Suivi ASN". La page d'édition n'a pas de metadata propre et hérite du titre générique du layout.
- **Correction :** Ajouter un `title.template` dans le layout root : `title: { default: "Suivi des accompagnements ASN", template: "%s — Suivi ASN" }`. Ajouter un `generateMetadata` pour la page d'édition.
- **Priorité :** 🟡 Mineur

---

**[NC] 10.11 — Pas de scroll horizontal à 320px de largeur**
- **Élément concerné :** Tableau "Tous les interlocuteurs" sur la page d'accueil
- **Problème :** Le tableau de données à 3 colonnes (Nom, Administration, Rôle) peut provoquer un défilement horizontal sur les viewports très étroits (320px). Le wrapper `.fr-table` du DSFR autorise le scroll horizontal pour les tableaux (acceptable pour les tableaux de données), mais l'expérience est dégradée sur mobile.
- **Correction :** Acceptable pour un tableau de données (exception RGAA 10.11). Pour améliorer l'UX mobile, envisager un affichage en cartes sous 320px. Priorité basse pour un outil interne desktop.
- **Priorité :** 🟡 Mineur

---

**[NC] 11.10 — Contrôle de saisie : messages d'erreur serveur**
- **Élément concerné :** Formulaires de création/modification de contact, formulaire d'interaction
- **Problème :** La validation côté serveur (zod) n'est pas restituée côté client avec `aria-invalid="true"` et `aria-describedby` pointant vers le message d'erreur. En cas d'échec de validation serveur (contournement du `required` HTML), l'utilisateur n'a pas de retour accessible. La validation native HTML (`required`) fonctionne mais ne couvre pas tous les cas (format email, etc.).
- **Correction :** Implémenter `useActionState` (React 19) pour capturer les erreurs serveur et les afficher champ par champ avec `state="error"` et `stateRelatedMessage` du composant DSFR `Input`.
- **Priorité :** 🟠 Majeur

---

**[NC] 11.13 — Autocomplete pour les champs d'interaction**
- **Élément concerné :** Formulaire d'ajout d'interaction (`interaction-form.tsx`)
- **Problème :** Le champ "Note" (textarea) n'a pas besoin d'autocomplete, mais le formulaire de contact a bien les bons `autoComplete` (`given-name`, `family-name`, `email`, `tel`). Cependant, le champ "Libellé" de la relance et le champ "Date de relance" ne sont pas des champs de données personnelles. Le champ email du formulaire de contact a `autoComplete="email"` ✓. Pas de NC sur les champs personnels. **En revanche**, le champ "Contexte d'accompagnement" (`textarea`) n'a pas de `autoComplete` — pas nécessaire car ce n'est pas une donnée personnelle normalisée. Finalement, ce critère est **conforme** pour les champs personnels mais le champ administration n'a pas `autoComplete="organization"`.
- **Correction :** Ajouter `autoComplete="organization"` sur le champ "Administration" du formulaire de contact.
- **Priorité :** 🟡 Mineur

---

**[NC] 12.1 — Deux systèmes de navigation minimum**
- **Élément concerné :** Navigation globale du site
- **Problème :** Le site ne dispose que d'un seul système de navigation global : le header avec lien vers l'accueil. La barre de recherche est présente uniquement sur la page d'accueil et cherche des interlocuteurs (pas des pages). Il n'y a pas de plan du site ni de menu de navigation listant les pages. Le fil d'Ariane (Breadcrumb) est présent sur les pages intérieures mais ne constitue pas un des systèmes listés par le RGAA (menu, plan du site, moteur de recherche).
- **Correction :** Ajouter un plan du site (`/plan-du-site`) accessible depuis le footer, ou ajouter un vrai moteur de recherche sur toutes les pages (dans le header par exemple).
- **Priorité :** 🟠 Majeur
